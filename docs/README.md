## .env 与 config.ini 的区别

### 结论：只使用 `.env`

**LightRAG 项目统一使用 `.env` 文件进行配置，`config.ini.example` 仅作为参考文档存在，不推荐使用。**

### 详细说明

LightRAG 支持三种配置方式（优先级从高到低）：

| 优先级 | 配置方式 | 说明 |
|--------|----------|------|
| **1（最高）** | 命令行参数 | `--llm-binding openai` 等，直接覆盖其他配置 |
| **2** | 环境变量 / `.env` 文件 | 启动时自动加载到系统环境变量 |
| **3（最低）** | `config.ini` | 仅用于存储配置（config-only），不推荐使用 |

**为什么只推荐 `.env`：**

1. **一致性**：`.env` 是项目约定俗成的配置方式，所有文档和示例都基于 `.env`
2. **可移植性**：`.env` 可通过 Docker、环境变量注入等方式灵活配置
3. **兼容性**：Docker Compose 中可直接设置环境变量覆盖 `.env`
4. **工具支持**：交互式配置向导（`make env-*`）只操作 `.env` 文件
5. **简洁性**：只需维护一个配置文件，无需同步 `config.ini`

**什么时候 `config.ini` 会有用：**

`config.ini` 主要用于存储外部数据库（如 Neo4j、MongoDB、PostgreSQL）的连接参数，提供了一种结构化的参考格式。示例文件 `config.ini.example` 列出了所有支持的存储后端配置项。

但实际上，这些参数同样可以通过 `.env` 中的环境变量来配置（参见 `env.example`），两者等价。

**使用示例：**

```bash
# .env 配置（推荐）
LLM_BINDING=openai
LLM_MODEL=gpt-4o-mini
EMBEDDING_BINDING=openai
EMBEDDING_MODEL=BAAI/bge-m3

# 命令行覆盖（最高优先级）
lightrag-server --llm-binding ollama --embedding-binding ollama
```

**结论：新建配置统一使用 `.env`，不要同时维护两个配置文件。**

## 文档更新规范

1. **修改代码时同步更新文档**：如果修改了 API、状态管理或工作流程，对应更新白皮书
2. **用户可见变更更新手册**：如果修改了 UI 交互，更新使用手册
3. **保持文档与代码同步**：定期检查文档中的路径、命令、参数是否与代码一致
4. **中文优先**：用户面向中文，优先使用中文编写文档
